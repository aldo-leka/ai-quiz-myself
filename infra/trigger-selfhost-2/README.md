# Trigger Self-Host 2

This folder contains the standalone Docker Compose setup that replaced our Coolify-managed Trigger deployment.

## Why We Moved

We started with Trigger running as a Coolify-managed service.

The symptoms that pushed us off that setup were:

- repeated gateway timeouts on prod website `trigger.example.com`
- production tasks stuck in `DEQUEUED`
- `supervisor` crash loops
- Coolify regenerating Trigger config and resetting critical env vars back to blank

The root config failures we found were blank workload API values:

```env
TRIGGER_WORKLOAD_API_PROTOCOL=
TRIGGER_WORKLOAD_API_PORT_EXTERNAL=
TRIGGER_WORKLOAD_API_DOMAIN=
```

That broke the stack in two stages:

1. `supervisor` refused to boot because protocol was blank
2. after protocol/port were fixed, runners still crashed because they built an invalid URL from an empty domain

The working values ended up being:

```env
TRIGGER_WORKLOAD_API_DOMAIN=supervisor
TRIGGER_WORKLOAD_API_PROTOCOL=http
TRIGGER_WORKLOAD_API_PORT_EXTERNAL=8020
```

We moved Trigger to plain Docker Compose because:

- config is explicit
- env is not regenerated by Coolify
- debugging is straightforward
- rollback is simpler
- runtime ownership is ours, not a generated Coolify service template

## What Still Comes From the Old Setup

This exact setup was cut over in a conservative way:

- Trigger runtime is no longer managed by Coolify
- the old Docker data volumes were reused
- the old Traefik proxy container (`coolify-proxy`) was reused for ingress/TLS

So this is the layout:

- app ownership: plain Docker Compose
- data: existing Docker volumes
- ingress: existing Traefik

## How We Booted It

1. Stop the old Coolify-managed Trigger resource.
2. Keep the old proxy running if you still want it to serve `trigger.example.com`.
3. Create a real `.env` from `.env.example`.
4. Start the standalone stack:

```bash
cd ~/trigger-selfhost-2
docker compose up -d
```

5. Verify health:

```bash
docker compose ps
docker compose logs --tail=200 webapp
docker compose logs --tail=200 supervisor
curl -ksS https://trigger.example.com/healthcheck
```

## Notes About Traefik

The `webapp` service includes Traefik labels and joins an external ingress network:

- `TRIGGER_INGRESS_NETWORK`
- `TRIGGER_DOMAIN`

That is what lets the standalone stack still answer on `trigger.example.com` while using the existing Traefik proxy container.

## If You Want Full Separation From Coolify

The next step after this setup is:

1. keep this compose setup
2. replace `coolify-proxy` with your own Caddy or Traefik
3. optionally move to new Docker volume names later

That is safer than migrating the data first.

## Cutover Strategy We Used

This was the safe path:

1. move runtime off Coolify
2. keep old data volumes
3. keep old Traefik temporarily
4. verify health and runner execution
5. later remove remaining Coolify dependencies

## Why This Worked

The working stack stopped relying on generated Coolify env and pinned the runner callback path explicitly:

- runners call `http://supervisor:8020`
- `supervisor` stays healthy
- `webapp` stays healthy
- `trigger.example.com` routes to the standalone `webapp`

That removed both the gateway timeout problem and the production runner startup failure caused by invalid workload API config.
