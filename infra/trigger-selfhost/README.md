# Trigger.dev on Coolify

This is a custom Trigger.dev self-host setup for Coolify, based on Trigger's official Docker stack, but adjusted to avoid the two problems you hit with the built-in Coolify Trigger resource:

1. the public app origin being wired to the autogenerated `sslip.io` `*_3000` URL
2. the database credentials being regenerated while persistent volumes still contain the old database cluster

## What this setup changes

- It uses the official Trigger webapp + supervisor containers.
- It pins the Trigger images to `v4.4.3`, which matches this app's current SDK version.
- It keeps fixed Postgres and ClickHouse credentials instead of autogenerated ones.
- It uses your real public URL for `APP_ORIGIN`, `LOGIN_ORIGIN`, and `API_ORIGIN`.
- It uses an external registry such as GHCR instead of a localhost registry, which is easier when deployments are triggered from your laptop.
- It keeps MinIO internal to the stack for object storage.
- It works around the current Trigger Docker entrypoint bug for plain `http://` ClickHouse URLs.
- It avoids custom Docker networks because Coolify documents those as a cause of `504 Gateway Timeout` for Compose services behind the proxy.

## Prerequisites

- Coolify proxy already working on your VPS.
- `trigger.example.com` DNS already pointing at your VPS.
- A GitHub account if you want to use GHCR.

## 1. Prepare the env file

Copy [`infra/trigger-selfhost/.env.example`](./.env.example) into Coolify's environment variable editor and replace every placeholder:

- `TRIGGER_PUBLIC_URL` must be `https://trigger.example.com`
- generate `TRIGGER_SESSION_SECRET`, `TRIGGER_MAGIC_LINK_SECRET`, `TRIGGER_ENCRYPTION_KEY`, and `TRIGGER_MANAGED_WORKER_SECRET` with:

```bash
openssl rand -hex 16
```

- keep `TRIGGER_POSTGRES_USER`, `TRIGGER_POSTGRES_PASSWORD`, `TRIGGER_POSTGRES_DB`, `TRIGGER_CLICKHOUSE_USER`, and `TRIGGER_CLICKHOUSE_PASSWORD` fixed
- set the GHCR values:
  - `TRIGGER_DOCKER_REGISTRY_URL=ghcr.io`
  - `TRIGGER_DOCKER_REGISTRY_NAMESPACE=<your GitHub username or org>`
  - `TRIGGER_DOCKER_REGISTRY_USERNAME=<your GitHub username>`
  - `TRIGGER_DOCKER_REGISTRY_PASSWORD=<GitHub token with package access>`

## 2. Create a custom Docker Compose resource in Coolify

Do not use the built-in Trigger resource template again.

In Coolify:

1. Create a new `Docker Compose` resource.
2. Paste [`infra/trigger-selfhost/docker-compose.coolify.yml`](./docker-compose.coolify.yml) into the compose editor.
3. Add the contents of [`infra/trigger-selfhost/.env.example`](./.env.example) as environment variables.
4. Save.

## 3. Attach the public domain to the webapp service

After the compose resource exists:

1. Open the `webapp` service inside that compose resource.
2. Set the service port to `3000`.
3. Add the domain as `https://trigger.example.com:3000`.
4. Enable HTTPS.

Browse to `https://trigger.example.com` normally. The `:3000` is only for Coolify's domain field so the proxy knows which container port to route to.

## 4. Deploy the stack

Deploy the compose resource.

The expected healthy services are:

- `webapp`
- `supervisor`
- `docker-proxy`
- `postgres`
- `redis`
- `electric`
- `clickhouse`
- `minio`

## 5. Verify the stack on the VPS

If you want to verify from the server:

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}'
curl -I https://trigger.example.com/healthcheck
curl -I https://trigger.example.com/login
```

The site should redirect to `/login`, and `/healthcheck` should return `200`.

## 6. First login

Open `https://trigger.example.com`.

If you have working SMTP or Resend configured, use the email login flow normally.

If `EMAIL_TRANSPORT` is blank, Trigger logs the magic-link URL in the `webapp` logs instead of sending it by email.

GitHub login is optional and is disabled by default. To enable it, set:

- `TRIGGER_AUTH_GITHUB_CLIENT_ID`
- `TRIGGER_AUTH_GITHUB_CLIENT_SECRET`

and create a GitHub OAuth app with callback URL:

```text
https://trigger.example.com/auth/github/callback
```

## 7. Set up the registry once

Trigger builds deployment images locally when self-hosting, so the machine that runs `trigger deploy` must be able to push to the registry.

On your laptop:

```bash
docker login ghcr.io -u <your-github-user>
```

If GHCR images end up private, also log in on the VPS host once:

```bash
docker login ghcr.io -u <your-github-user>
```

## 8. Create the Trigger project and API keys

Inside the Trigger dashboard:

1. Create a new project.
2. Copy its project ref.
3. Copy the `DEV` or `PROD` secret key you want your app to use.

Then update your local app:

- replace the project ref in [`trigger.config.ts`](../../trigger.config.ts)
- replace `TRIGGER_SECRET_KEY` in your local `.env`
- add `TRIGGER_API_URL=https://trigger.example.com` to your local `.env`

## 9. Log the CLI into the self-hosted instance

Your app currently uses `@trigger.dev/sdk` `4.4.3`, so keep the CLI on the same version when possible.

```bash
npx trigger.dev@4.4.3 login -a https://trigger.example.com --profile self-hosted
npx trigger.dev@4.4.3 whoami --profile self-hosted
```

If `whoami` still shows `https://api.trigger.dev`, you are still on your cloud default profile. Switch to the self-hosted profile:

```bash
npx trigger.dev@4.4.3 switch self-hosted
npx trigger.dev@4.4.3 whoami
```

You can test the self-hosted instance locally without deploying first:

```bash
npx trigger.dev@4.4.3 dev --profile self-hosted
```

If you have already switched the default CLI profile to `self-hosted`, this should also work:

```bash
npm run trigger:dev
```

After local dev is working, deploys can go to your self-hosted instance:

```bash
npx trigger.dev@4.4.3 deploy --profile self-hosted --self-hosted --push
```

For CI, set:

- `TRIGGER_API_URL=https://trigger.example.com`
- `TRIGGER_ACCESS_TOKEN=<token from the self-hosted instance>`

## 10. If you need to reset the stack

Only do this if the stack is disposable and you do not care about the data yet.

```bash
cd /path/to/the-compose-project
docker compose down -v --remove-orphans
docker compose up -d
```

If you ever change `TRIGGER_POSTGRES_USER` or `TRIGGER_POSTGRES_PASSWORD` after the first successful boot, you can reproduce the `electric is unhealthy` failure you already saw.
